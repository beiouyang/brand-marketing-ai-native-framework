# 针对「国补&以旧换新双重优惠弹层」的设计建议

> 生成时间：2026-05-27  
> 知识库引用：国补叠加以旧换新利益点感知提升 · 国补到手价 · App国补多商品

---

## 一、需求解读

**要解决的问题：** 用户在优惠弹层中看到国补和以旧换新两项优惠分散展示，无法直观感知叠加后的总优惠金额，需要自行心算组合。

**业务目标：** 通过优惠弹层的设计优化，强化用户对双重优惠的感知，降低用户决策成本，促进订单转化。

**设计挑战：** 在弹层的有限空间内，既要分别展示两项优惠的独立信息（各自金额、条件、有效期），又要聚合突出叠加后的最终到手价；同时需处理多种状态组合（国补已领/未领、有旧机/无旧机）而不互相混淆。

**成功标准：** 用户打开优惠弹层，一眼就能看到「国补减 xxx + 以旧换新再减 xxx = 到手价 xxx」的叠加结论，不需要拼接认知。

---

## 二、设计建议

**核心方向：** 优惠弹层采用「聚合 → 详情」的双层结构：顶部优先展示双项叠加的最终到手价（核心结论），下方分别展示国补和以旧换新的独立优惠明细（支撑数据）。通过颜色体系和信息层级的区分，让用户快速定位到「我能省多少钱」，然后按需查看具体规则。

---

### 建议1：弹层顶部设置「叠加利益点汇总卡片」，聚合展示双项到手价

**结构：**

```
┌─────────────────────────────────────┐
│  国补 + 以旧换新叠加利益点          │
│                                     │
│  国补再减 ¥500 + 以旧换新再减 ¥800 │  ← 绿色 + 服务金色并列
│            ↓                        │
│          到手价 ¥1300               │  ← 大号、黑色、加粗
│       (节省 ¥1300)                  │  ← 可选：对比价格锚点
│                                     │
│  [查看详情 ▼]                      │  ← 引导展开下方详情
└─────────────────────────────────────┘
```

**原因：** 聚合卡片占据弹层顶部最显眼的位置，用户第一眼落在「到手价 ¥1300」这个结论，直接解答了「我能省多少钱」。数字使用大号字体（推荐 24-28px）和品牌主色，确保视觉权重最高。两项减额分别用对应的颜色标识（绿色 = 国补政策色，服务金色 = 以旧换新权益色），避免色混乱。

**设计约束：** 
- 仅在「国补已领 + 以旧换新满足条件」时显示完整卡片
- 若国补未领取，此卡片不展示；改为展示「国补可减 xxx，[立即领取]」的引导卡片
- 若以旧换新无旧机，卡片中以旧换新部分显示「预估减 xxx」（灰色文案），而非「再减」

---

### 建议2：下方分行展示国补和以旧换新的独立优惠，但优化排序与视觉权重

**结构（国补已领 + 有旧机场景）：**

```
┌────────────────── 汇总卡片 ──────────────────┐
│ 到手价 ¥1300                                 │
└────────────────────────────────────────────┘

├─ 国家补贴（政策支持）
│  ├─ 京东支付再减 ¥500
│  ├─ 有效期：2026-06-30 23:59 前
│  └─ [补贴说明 i]  ← 轻量问号图标，点击触发说明弹窗
│
├─ 以旧换新（权益补贴）
│  ├─ 旧机估价 ¥2000，可再减 ¥800（换新到手价）
│  ├─ 有效期：2026-06-30 23:59 前
│  └─ [修改旧机 >]  ← 快捷入口重新选择
│
└─ 其他优惠（可选）
   └─ [更多可享优惠 >]
```

**原因：** 
- **排序优先级**：国家补贴置顶（政策属性优于权益属性）。这符合用户对「政府的钱」的重视程度，也避免了与历史 App 国补多商品方案中已享优惠混排的问题。
- **分行而非混排**：避免「国补减 ¥500 + 以旧换新减 ¥800」这种混排方式，用户容易漏看。分行后每项独占一个视觉区块，清晰明确。
- **有效期并列显示**：两项有效期可能不同，分别标注避免误解。
- **补贴说明入口**：国补行右侧添加轻量问号图标，点击触发「应补贴政策要求，部分附件不可享国补…」的信息弹窗（295px 非全宽），符合渐进式政策说明的原则。

### 建议3：处理状态转换时的实时更新机制

当用户在弹层内点击「立即领取」国补时，弹层需要支持「无需关闭，直接更新」：

**用户交互流程：**

```
初始态（未领国补）       →  用户点击"领取"  →  弹层局部更新
├─ 汇总卡片显示：          ├─ 调用领取接口     ├─ 汇总卡片出现
│  「国补可减 xxx，            │   │             │  「国补再减 xxx」
│   以旧换新再减 xxx」        │   └─ 前端状态    │
│                              │     变更         ├─ 国补行从「可减」
├─ 国补行显示：               │                 │  变更为「已减」
│  「可减 ¥500 [立即领取]」│                 │
│                            │              └─ 避免关闭重新打开
├─ 以旧换新行显示：         │
│  「换新到手价 ¥800」      │
│                            │
└─ [修改旧机]              │
```

**原因：** 领取后无需关闭弹层重新打开，用户可直接看到新的到手价，降低交互成本。这要求弹层内的国补行和汇总卡片具备「状态响应」能力。

**技术实现提示：** 领取接口回调后，前端仅需更新国补的状态字段（如 `subsidy_status: 0→2`），汇总卡片和国补行的显示会自动联动切换，无需刷新整个弹层。

---

### 建议4：严格区分「到手价」vs「预估价」的两套文案

当国补已领但以旧换新无旧机时，弹层应显示：

```
汇总卡片（降级）：
├─ 国补再减 ¥500（确定）
└─ 以旧换新预估减 ¥800（灰色文案，表示不确定）
   = 预估到手价 ¥1300（灰色显示）

下方详情行：
├─ 国补行：「京东支付再减 ¥500」
└─ 以旧换新行：「预估可减 ¥800（选定旧机后确认）」
   └─ [选择旧机 >]  ← 快捷入口
```

**原因：** 
- 用户容易错误地认为「预估减 ¥800 = 我已经能拿到」。严格用「预估」vs「再减」的文案差异来区分已确定 vs 待确认。
- 灰色视觉处理强化「待确认」的信号，与黑色的已确定金额形成视觉对比。
- 「预估到手价」而非「到手价」，避免混淆。

---

### 建议5：处理多种状态组合的完整覆盖

弹层需要支持以下状态的无缝切换（不同用户流程）：

| 状态组合 | 汇总卡片展示 | 国补行 | 以旧换新行 | 风险提示 |
|---|---|---|---|---|
| 已领 + 有旧机 | ✅ 显示聚合「到手价」 | 「再减 ¥500」 | 「再减 ¥800」 | — |
| **已领 + 无旧机** | ⚠️ 「预估到手价」（灰色） | 「再减 ¥500」 | 「预估 ¥800」（灰色） | **易遗漏**：研发可能整行隐藏，建议保留「预估」以提示用户还需选机 |
| 未领 + 有旧机 | ⚠️ 仅显示以旧换新「再减 ¥800」+ 国补「可减 ¥500」（引导态） | 「可减 ¥500 [立即领取]」 | 「再减 ¥800」 | — |
| 未领 + 无旧机 | ⚠️ 「预估到手价」两项都灰色 | 「可减 ¥500 [立即领取]」 | 「预估 ¥800」（灰色） | **易遗漏**：用户无旧机时，以旧换新行容易被隐藏 |
| 无旧机，已切换出以旧换新 | 💔 回退显示纯国补「再减 ¥500」 | 「再减 ¥500」 | 行隐藏 | 需确认与产品的降级策略 |

## 三、设计边界

### 需要覆盖的状态

| 状态 / 场景 | 视觉处理 | 风险提示 |
|---|---|---|
| ⭐ 国补已领 + 以旧换新有旧机 | 汇总卡片展示「国补再减 + 以旧换新再减 = 到手价」，两项都是黑色确定态 | 这是理想状态，优先设计 |
| ⭐ 国补已领 + 以旧换新无旧机 | 汇总卡片降级为「预估到手价」，以旧换新部分用灰色表示待选择 | **高风险遗漏**：研发容易把「无旧机」的以旧换新行完全隐藏，建议设计时显式标注「预估」 |
| 国补未领 + 有旧机 | 汇总卡片仅显示以旧换新的「再减」，国补部分显示「可减 xxx [立即领取]」 | — |
| 国补未领 + 无旧机 | 两项都是「可减/预估」的引导态，汇总卡片不显示最终结论 | — |
| 无国补资格 | 不展示国补行，仅展示以旧换新优惠 | — |
| 线下门店降级 | ⚠️ TODO: 需确认是否隐藏在线国补领取入口，或降级为「不支持线下门店」的灰态 | 需与产品确认降级策略 |

### 动笔前需要确认的问题

| # | 问题 | 不确认的后果 | 找谁 |
|---|---|---|---|
| 1 | 国补未领时，汇总卡片是否显示两项优惠的聚合预估值？还是仅显示「可领 xxx」？ | 直接影响弹层的首屏设计和用户对总优惠的第一印象 | 产品 |
| 2 | 用户在弹层内点击「领取国补」后，弹层是否需要实时更新？还是关闭后重新打开才生效？ | 决定前端的状态管理复杂度和用户交互满意度 | 产品 + 前端研发 |
| 3 | 以旧换新无旧机时，该行是否保留「预估减 xxx」的占位提示？还是完全隐藏？ | 决定用户是否能意识到「还可以通过选机来增加优惠」 | 产品 + UI |
| 4 | 是否需要在优惠弹层支持「横滑容器」？（优惠条目过多时） | 影响弹层的整体高度和多优惠的排列策略 | 产品 |
| 5 | 聚合卡片中「节省 xxx」的对比锚点是参考什么？原价？近 90 天最低价？ | 影响用户对价格力的感知，可能存在误导的法律风险 | 产品 + 法务 |

## 四、历史依据

### 🔴 国补叠加以旧换新利益点感知提升

**与本需求的关联：** 直接的多优惠聚合展示方案，已覆盖三态状态机和优惠弹层内的叠加设计。

**可直接沿用的决策：**

- **Decision 3：已领取态下，国补叠加以旧换新利益点聚合展示** — 在用户已领取国补资格且商品支持以旧换新时，在价格楼层区域聚合展示「国补减 xxx + 以旧换新再减 xxx = 叠加到手价 xxx」的利益点组合。两项优惠以视觉区块的形式并排或层叠展示，颜色使用服务金（ServiceGolden）体系区分于纯国补的绿色语义。

- **Pattern 1：三态状态机驱动的促销模块切换** — 后端返回用户状态字段（0 = 无资格 / 1 = 有资格未领取 / 2 = 已领取），前端对应渲染三套腰带 + 价格楼层模板。用户完成领取后，页面无需刷新，由前端状态变更触发局部更新。核心原则：状态机切换的边界要清晰，每种状态对应唯一的视觉方案，禁止状态之间的视觉混用。

- **Pattern 2：多优惠聚合呈现** — 确认用户已满足所有优惠条件（均已激活 / 领取），在核心利益点位置聚合展示「项目A减xxx + 项目B减xxx = 到手价xxx」。核心原则：聚合展示的是「已确定可拿到」的到手价，不是「最高可能」的估算值；估算值应用「最高减」表达，确定值用「再减」或「到手价」表达。

**本次是否需要调整：** 沿用模式，但优惠弹层是比价格楼层更下沉的信息层，需要额外处理「未领」状态下的引导和预估价的灰态展示。

### 🔴 国补到手价

**与本需求的关联：** 优惠弹层的"到手价 vs 预估价"严格区分规则，以及多优惠分行展示的交互处理。

**可直接沿用的决策：**

- **Decision 1：以旧换新场景严格区分『到手价』与『预估价』** — 购新场景：已领取 → 「国补换新到手价」；未领取 → 「国补换新预估价」。以旧换新场景：有资格已领取 + 有旧机 → 「国补换新到手价」；其余所有状态（已领取无旧机 / 有资格未领取）→ 「国补换新预估价」。

- **Decision 2：优惠弹层超出屏宽时横滑，不折叠** — 优惠条目（国家补贴/满减/京东支付立减/云闪付立减/政府消费券）在弹层内横向排列。条目总宽度 ≤ 屏宽时居中展示；超出时支持左右横滑。

**本次是否需要调整：** 沿用「到手价 vs 预估价」的文案策略。在优惠弹层中，需要用灰色视觉处理强化「预估」的待确认信号。

### 🟡 App国补多商品

**与本需求的关联：** 优惠弹层内国补优惠行的基础设计（粉色卡片、补贴说明弹窗等），可作为参考。

**可直接沿用的决策：**

- **Decision 3：优惠弹层内国补行增加附件提示，触发国补说明弹窗** — 优惠弹层的信息模块从上到下依次为：到手价计算、已享优惠、附件、更多可享优惠。在已享优惠模块的国补优惠行下方，增加附件提示文案，用户点击触发国补说明弹窗（295px 非全宽）。

- **Pattern 2：渐进式政策说明（两层信息披露）** — 主页 / 弹层：一句话提示。用户主动点击 → 信息型弹窗（政策解释）。弹窗内引导链接 → 详情弹层（具体数据）。

**本次是否需要调整：** 沿用渐进式说明的框架。优惠弹层的国补行旁边添加轻量问号图标，点击触发补贴说明弹窗，避免在弹层内堆砌过多政策文案。

## 附录：设计要点速查表

### 颜色规范

| 元素 | 颜色值 | 用途 |
|---|---|---|
| 国补政策色 | #0AAD48 | 国补减额、政策类权益 |
| 以旧换新权益色（服务金） | #b5691a（文字）/ #ffd4a3（背景） | 以旧换新减额、服务权益 |
| 确定态文字 | #1a1a1a（黑色） | 已确定的到手价、已减额 |
| 预估态文字 | #888888（灰色） | 预估减额、待确认价格 |
| 价格锚点强调 | #FF0F23（品牌红） | 有效期提示、关键数字 |

### 交互核心规则

1. **聚合卡片始终在弹层顶部** —— 用户第一眼的落点必须是「我能省多少钱」
2. **国补未领时，不展示聚合结论** —— 改为展示引导状态「国补可减 xxx [立即领取]」
3. **以旧换新无旧机时，保留预估行** —— 用灰色表示「还未确定，可通过选机确认」
4. **领取后实时更新** —— 弹层内点击领取后无需关闭重新打开，直接更新显示
5. **补贴说明用轻量弹窗** —— 295px 非全宽，避免阻挡主弹层背景

### 文案规范

| 场景 | 文案示例 | 说明 |
|---|---|---|
| 国补已领 | 「京东支付再减 ¥500」 | 「再减」表示叠加在其他优惠基础上的确定减额 |
| 国补未领 | 「京东支付最高减 ¥500」 | 「最高」表示潜在上限，激励用户领取 |
| 预估价格 | 「换新预估价 ¥1300」 | 明确告知非最终价格，防止误解 |
| 确定价格 | 「换新到手价 ¥1300」 | 「到手」表示已确定的最终减额 |
| 补贴说明 | 「应补贴政策要求，部分附件不可享受国补…」 | 在信息弹窗中展示，不在主弹层中 |
