# 针对「国补商品搭配组合优惠」的设计建议

---

## 一、需求解读

**要解决的问题：** 用户购买国补商品时，无法直观看到该商品与其他商品组合购买能享受的叠加优惠，缺乏联动购买的决策依据。

**业务目标：** 借助国补商品的引流优势，将国补商品与非国补关联商品打包为组合优惠，带动非国补商品销售，拓宽营收边界。

**设计挑战：** 组合内商品可能有不同的补贴状态（全部未领/部分未领/全部已领/全部已用），需要在有限的空间内清晰传达每件商品的补贴进度和组合后的到手价，同时不增加用户的认知负担。

**成功标准：** 用户一眼能看出「组合购买比单买省多少」，知道哪些补贴已领、哪些还需操作，并能顺畅完成组合购买。

## 二、推荐参考的设计稿

### 🔴 最强参考：国补支持新虚拟组套

**为什么参考这个：** 这是最直接相关的历史 Case，完整实现了「商详主页组套楼层 → 套装详情弹层 → 补贴领取 → 组合购买」的完整流程，框架可直接复用。

**打开设计稿：** [Relay 链接](https://relay.jd.com/file/design?id=2046484439810203649&page_id=6:44579&node_id=6:49261)

**可以直接复用的部分：**
- **组套楼层框架**：商品缩略图+价格行+补贴状态摘要的楼层布局，无需重新设计
- **套装详情弹层结构**：弹层内商品卡排列方式、补贴状态标签样式
- **四态分层逻辑**：全部未领/部分未领/全部已领/全部已用的状态切换框架
- **底部工具栏联动**：底部按钮随补贴领取进度动态变化的逻辑

**需要在此基础上修改的部分：**
- **组合商品数量适配**：历史 Case 支持 2件/5件两种规格 → 本次需确认组合商品数量上限，可能需适配更多或更少的规格
  → 参考：[虚拟组套 Case §二、设计核心打法]
- **非国补商品价格行**：历史 Case 中非国补商品仅展示常规价格 → 本次需明确非国补商品在组合中的优惠来源和展示方式
  → 参考：[虚拟组套 Case §三、状态枚举·商品卡]

**本次需要全新设计的部分：**
- **组合优惠的定价展示**：历史 Case 展示的是「补贴后最低到手价」→ 本次需设计「组合优惠价 vs 单买总价」的对比展示，突出「组合省多少」
  → 可参考的设计模式：[到手价 Case Pattern 1「预估价→到手价」切换]
  → ⚠️ 设计师需要补充：组合优惠的定价规则（固定折扣/满减/叠加补贴）
  → ❓ 需要先确认：组合优惠是否与国补补贴叠加？还是二选一？

## 三、设计任务清单

**直接复用（无需设计）：**
- [ ] 组套楼层基础框架 — 沿用虚拟组套 Case 的楼层布局，无需改动
- [ ] 套装详情弹层结构 — 沿用虚拟组套 Case 的弹层框架
- [ ] 四态分层逻辑 — 沿用虚拟组套 Case 的全部未领/部分未领/全部已领/全部已用

**修改现有设计：**
- [ ] 组合商品数量适配 — 根据实际商品数量上限调整楼层展示规格，基于虚拟组套 Case [Relay 链接]
- [ ] 非国补商品价格行 — 明确非国补商品在组合中的价格展示方式
- [ ] 底部工具栏按钮文案 — 根据组合状态动态切换（「领取全部补贴」/「组合购买」/「已全部领取」）

**全新设计：**
- [ ] 组合优惠价 vs 单买总价的对比展示 — 突出「组合省多少」的核心利益点
- [ ] 组合优惠的入口引导文案 — 在商详主页引导用户进入组合楼层
- [ ] 组合优惠的兜底状态 — 当组合内所有补贴已用尽时的展示

**暂不设计（本次排除）：**
- 以旧换新叠加场景 — 待确认组合优惠是否支持以旧换新叠加
- 结算页的组合优惠展示 — 结算页独立设计，本次不涉及
- PC 端适配 — 本次为 App 端专属

## 四、需要覆盖的状态

| 状态 / 场景 | 视觉处理方向 | 风险提示 |
|---|---|---|
| 全部未领 | 所有商品补贴均未领取，展示「领取全部补贴」引导按钮 | — |
| ⭐ 部分未领 | 混合展示已领/未领状态，区分商品行样式（已领→绿色标签，未领→灰色标签+领取按钮） | 来自 [虚拟组套 Case]，历史曾遗漏 |
| 全部已领 | 所有补贴已领，展示组合最低到手价 | — |
| ⭐ 全部已用 | 补贴已全部使用的兜底终态，展示「已全部使用」文案 | 来自 [虚拟组套 Case]，历史易遗漏的终态 |
| 组合内无国补商品 | 所有商品均为非国补商品，楼层降级为普通组合推荐 | ⚠️ 需与产品确认是否覆盖 |
| 组合楼层无腰带态 | 国补腰带不存在时的楼层兜底展示 | — |
| ⚠️ 暗黑模式 | 楼层和弹层的暗黑模式适配 | 需确认本期是否覆盖 |

### 动笔前需要确认的问题

| # | 问题 | 不确认的后果 | 找谁 |
|---|---|---|---|
| 1 | 组合内商品数量上限是多少？ | 楼层布局无法确定，影响设计稿输出 | 产品 |
| 2 | 组合优惠的定价规则是什么？（固定折扣/满减/叠加补贴） | 无法确定到手价的计算方式和展示逻辑 | 产品 |
| 3 | 非国补商品在组合中的优惠来源是什么？ | 无法确定非国补商品的价格行展示内容 | 产品 |
| 4 | 组合优惠是否支持以旧换新叠加？ | 可能遗漏以旧换新场景的状态覆盖 | 产品 |
| 5 | 组合优惠的入口位置在哪里？（商详页哪个楼层插入） | 楼层在商详页的插入位置不确定，影响页面布局 | 产品 |

## 五、历史依据摘要

> 以下决策来自历史 Case，可直接沿用，无需重新讨论。

### 🔴 国补支持新虚拟组套

**可直接沿用的决策：**
> 在商详页新增「虚拟组套」楼层模块，将 1 件国补商品与 1～4 件非国补关联商品打包，通过套装详情弹层承载补贴领取和组套购买流程。 — 来源：[虚拟组套 Case §二、设计核心打法]

> 商详楼层是流量最集中的曝光位，在此新增组套入口可最大化带动非国补商品的连带购买；以弹层形式承载套装详情，保持商详主页结构不变，降低对既有楼层布局的干扰。 — 来源：[虚拟组套 Case §二、设计核心打法]

> 四态分层：全部未领 → 部分未领 → 全部已领 → 全部已用，每一态都要给用户一个明确的下一步行动指引。 — 来源：[虚拟组套 Case §附：可复用的设计模式·Pattern 1]

### 🟡 App 国补多商品

**可直接沿用的决策：**
> 领取前文案「京东支付最高减 xxx」，领取后文案「京东支付再减 xxx」——「最高减」传达潜在上限激励领取，「再减」表达已到手确定感。 — 来源：[多商品 Case §4·Decision 2]

> 国补腰带作为商详主页的国补入口，独立于已享楼层，绿色（#0AAD48）为国补活动专属色。 — 来源：[多商品 Case §4·Decision 1]

### 🟡 国补到手价

**可直接沿用的决策：**
> 「预估价」和「到手价」的文案切换必须严格对应状态：未满足条件时展示「预估价」，用户完成前置操作后切换为「到手价」。 — 来源：[到手价 Case §8·Pattern 1]

> 优惠条目超出屏宽时横滑，不折叠。 — 来源：[到手价 Case §4·Decision 2]
