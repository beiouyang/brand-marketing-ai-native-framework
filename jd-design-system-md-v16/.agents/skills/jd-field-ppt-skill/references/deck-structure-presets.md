# Deck 结构化路径

参考 `feishu-cli-board` 的多路径思路：先判断“这份材料要解决什么表达问题”，再选择 PPT 结构路径。不要默认套同一套封面 + 摘要 + 链路。

## 选型决策树

```
材料要表达什么？
│
├─ 讲用户如何完成一件事
│  ├─ 触点多 / 决策链长 → 路径 A：用户旅程型
│  └─ 只需快速总览 → 路径 A1：一页旅程总览
│
├─ 讲系统/算法/能力如何运转
│  ├─ 有输入、处理、输出、反馈 → 路径 B：系统机制型
│  └─ 有模块分层和依赖关系 → 路径 B1：架构分层型
│
├─ 讲业务复盘或增长判断
│  ├─ 有指标和归因 → 路径 C：经营复盘型
│  └─ 有实验结论 → 路径 C1：实验决策型
│
├─ 讲产品方案或策略提案
│  ├─ 需要争取共识和资源 → 路径 D：方案提案型
│  └─ 需要比较多个选择 → 路径 D1：对比决策型
│
├─ 讲设计系统/规范发布
│  ├─ 需要团队按规则执行 → 路径 E：规范发布型
│  └─ 需要展示组件结构 → 路径 E1：组件解剖型
│
└─ 信息量很大但只能一页
   └─ 路径 F：一页压缩型
```

## 路径 A：用户旅程型

适合：用户故事、购买链路、服务链路、触点 Hints、商家经营路径。

推荐页序：

1. `cover-field`：一句主判断 + 用户故事 + 目标
2. `journey-lane`：3-6 个触点，标题写动作或决策变化
3. `anatomy-system`：背后的数据/模型/机制
4. `kpi-strip` 或 `closing-action`：目标指标或下一步

压缩为一页时：

- 上方：主判断 + 目标
- 中部：主链路 5 步以内
- 下方：机制底座或关键输入
- 底部：延展/风险/下一步一句话

## 路径 B：系统机制型

适合：算法框架、推荐链路、LLM 生成、排序、反馈闭环、数据管道。

推荐页序：

1. `cover-field`：系统要解决的业务问题
2. `anatomy-system`：输入 → 处理 → 输出
3. `journey-lane` 或 `roadmap`：线上链路 / 迭代链路
4. `evidence-table`：模型指标、策略约束、验证结果

表达规则：

- 输入不超过 7 类。
- 处理层最多 3 段：生成、排序、编排 / 治理。
- 输出必须落到具体触点或业务动作。
- 反馈闭环必须说明如何回到模型或策略。

## 路径 C：经营复盘型

适合：GMV、转化、留存、搜索、内容增长、频道经营、活动复盘。

推荐页序：

1. `cover-field`：一句经营判断
2. `executive-summary`：3 个结论
3. `kpi-strip`：核心指标
4. `evidence-table`：证据和归因
5. `roadmap`：下一步动作

表达规则：

- 标题必须是结论句。
- 指标必须有口径、时间和单位。
- 不把图表做成装饰，图表只服务判断。

## 路径 D：方案提案型

适合：产品方案、策略升级、资源申请、跨团队共识。

推荐页序：

1. `cover-field`：要改变什么
2. `executive-summary`：机会、方案、收益
3. `compare-before-after`：现状和目标
4. `anatomy-system`：方案结构
5. `risk-control`：风险和兜底
6. `roadmap`：分阶段落地

## 路径 E：规范发布型

适合：设计系统、组件规范、交互规则、内容规范、AI 生成规则。

推荐页序：

1. `cover-field`：规范发布目的
2. `agenda-bands`：定义 / 原则 / 类型 / 场景
3. `anatomy-system`：结构拆解
4. `compare-before-after`：正反例或旧新对比
5. `evidence-table`：规则清单
6. `closing-action`：执行要求

## 路径 F：一页压缩型

适合：评审会、汇报附录、战略会快速过方案、原始图信息很密但只能一页。

布局顺序：

1. 顶部 20%：主判断 + 目标卡
2. 中部 45%：主链路或主框架
3. 下部 25%：机制底座 / 反馈闭环 / 关键证据
4. 底部 10%：结论、下一步或口径说明

压缩规则：

- 每个卡片最多 1 个标题 + 1 句说明 + 1 个标签。
- 主链路不超过 5 步；系统分段不超过 3 段。
- 次级信息进小字脚注或附录，不在一页里硬塞。
- 如果文字必须小于 12px 才放得下，改为多页。

## 路径选择速查

| 用户材料 | 推荐路径 |
|---|---|
| “用户从首页到下单/售后怎么被推荐 Hints” | A 或 A1 |
| “引流语生成、排序、分发、反馈框架” | B 或 F |
| “Q2 经营复盘 / 指标变化 / 下一步动作” | C |
| “我要讲一个新产品能力或策略方案” | D |
| “组件规范 / 设计系统发布 / 规则说明” | E |
| “只能做成一页” | F |

## 兜底判断

- 如果用户没有指定页数，优先多页结构，保证可讲述。
- 如果用户要求“一页”，优先路径 F，主动压缩文案。
- 如果材料像架构图但听众是业务方，先用 D 或 C，不要直接上 B。
- 如果材料像用户流程但核心诉求是模型能力，先用 A 讲故事，再用 B 讲机制。
